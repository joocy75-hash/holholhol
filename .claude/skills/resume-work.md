# 작업 재개 (Resume Work)

이 스킬은 토큰 소진으로 중단된 작업을 새 계정에서 재개할 때 사용합니다.

## 실행 순서

### 1단계: 저장소 업데이트
가장 먼저 최신 코드를 받습니다.

```bash
cd /Users/mr.joo/Desktop/holdem
git pull origin develop
```

### 2단계: 상태 파일 읽기
중단 전 저장된 상태 파일을 확인합니다.

**읽어야 할 파일**:
- `.planning/current_state.md` - 현재 작업 상태
- `docs/WORK_PLAN_2026.md` - 전체 작업 계획 및 완료 체크

### 3단계: 진행 상황 분석

상태 파일에서 다음 정보를 추출합니다:
- **진행 중인 Phase**: 어떤 Phase를 작업 중이었는지
- **완료율**: 해당 Phase의 몇 %까지 완료했는지
- **수정 중인 파일**: 어떤 파일들을 수정하고 있었는지
- **다음 작업**: 다음에 해야 할 작업 목록
- **주의사항**: 특별히 주의해야 할 사항

### 4단계: 컨텍스트 요약

사용자에게 다음 정보를 요약해서 보고합니다:

```markdown
## 🔄 작업 재개 준비 완료

### 📊 이전 작업 상태
- **Phase**: [Phase 번호 및 이름]
- **진행률**: [X%]
- **마지막 작업**: [마지막으로 한 작업]

### 📝 수정 중인 파일
1. [파일명 1] - [수정 내용 요약]
2. [파일명 2] - [수정 내용 요약]

### ✅ 다음 단계
1. [ ] [다음 작업 1]
2. [ ] [다음 작업 2]
3. [ ] [다음 작업 3]

### ⚠️ 주의사항
- [주의사항 1]
- [주의사항 2]

---

**작업을 재개하시겠습니까?** (y/n)
```

### 5단계: 사용자 확인 후 작업 재개

사용자가 확인하면 다음 작업을 바로 시작합니다:
- 서브에이전트가 필요한 경우 자동으로 호출
- 수정 중인 파일 열기
- 다음 단계 시작

## 예시

### 입력
```
/resume-work
```

### 출력
```markdown
## 🔄 작업 재개 준비 완료

### 📊 이전 작업 상태
- **Phase**: Phase 1.1 - 헤즈업(2인) 게임 규칙 구현
- **진행률**: 60%
- **마지막 작업**: `_get_next_acting_player()` 메서드 수정 중

### 📝 수정 중인 파일
1. backend/app/game/poker_table.py (Line 450-520)
   - 헤즈업 액션 순서 로직 추가 중
2. tests/unit/game/test_heads_up.py (새 파일)
   - 3개 테스트 케이스 작성 완료

### ✅ 다음 단계
1. [ ] 1.1.4 헤즈업 ↔ 멀티 전환 처리 완료
2. [ ] 1.1.5 단위 테스트 7개 추가
3. [ ] pytest 실행하여 검증

### ⚠️ 주의사항
- `_is_heads_up_mode` 플래그로 모드 전환 감지
- 기존 3인 이상 게임 로직은 유지해야 함

---

저장소가 최신 상태로 업데이트되었습니다.
다음 작업(1.1.4)을 시작하시겠습니까? (y/n)
```

## 중요 체크포인트

### 상태 파일이 없는 경우
`.planning/current_state.md` 파일이 없으면:

1. `docs/WORK_PLAN_2026.md`에서 마지막 ✅ 체크된 항목 찾기
2. 다음 미완료 항목을 "다음 작업"으로 제안
3. 사용자에게 어디서부터 시작할지 물어보기

### Git 충돌이 있는 경우
`git pull` 시 충돌 발생하면:

1. 충돌 파일 목록 보여주기
2. `git status` 결과 출력
3. 사용자에게 수동 해결 요청
4. 해결 후 이 스킬 재실행 안내

## 에러 처리

### 파일 읽기 실패
상태 파일 읽기 실패 시:
```
⚠️ 상태 파일을 찾을 수 없습니다.
대신 WORK_PLAN_2026.md에서 마지막 완료 항목을 확인합니다...
```

### Git 업데이트 실패
Git pull 실패 시:
```
❌ Git 업데이트 실패. 네트워크를 확인하거나 수동으로 pull을 실행해주세요.
git pull origin develop
```

## 사용 시나리오

### 시나리오 1: 정상적인 재개
```
사용자: /resume-work
AI: [상태 파일 읽음]
AI: [진행 상황 요약 출력]
사용자: y
AI: [다음 작업 시작]
```

### 시나리오 2: 상태 파일 없음
```
사용자: /resume-work
AI: 상태 파일이 없습니다. WORK_PLAN_2026.md를 확인합니다...
AI: 마지막 완료: Phase 1.1
AI: 다음 작업: Phase 1.2를 시작하시겠습니까?
사용자: y
AI: [Phase 1.2 시작]
```

### 시나리오 3: 특정 Phase 지정
```
사용자: /resume-work 1.3
AI: Phase 1.3 환불 로직 구현을 시작합니다...
AI: [Phase 1.3 작업 내용 출력]
```

## 주의사항

1. **항상 Git pull 먼저**: 최신 코드 확인 필수
2. **상태 파일 신뢰**: 상태 파일 내용을 그대로 따름
3. **사용자 확인 필수**: 자동으로 작업 시작하지 말고 확인 받기
4. **서브에이전트 적극 활용**: 복잡한 작업은 에이전트에게 위임

## 관련 스킬

- `/save-state`: 작업 상태 저장 (토큰 소진 전)
- `/check-progress`: 현재 진행 상황 확인
